# Forge Distribution and Upgrade Policy (E9-S4)

Status: Active (RC/Beta)
Last Updated: 2026-02-28
Owner: Platform

## Scope

This policy defines how Forge release artifacts are distributed, validated, upgraded, rolled back, and deprecated across release channels.

## Distribution Strategy

1. Primary distribution artifact is a signed release tarball generated by `.github/workflows/forge-release.yml`:
   - `artifacts/forge-<channel>-<version>-<sha>.tar.gz`
   - `artifacts/release.sha256`
   - `artifacts/release.sha256.asc`
2. Workspace packages remain private until a separate package-publication decision is approved.
3. Channel assignment is enforced at release time via `scripts/verify-distribution-policy.mjs`.

## Release Channels

1. `stable`
   - Target: production users.
   - Version rule: non-prerelease semantic versions only (for example `1.4.2`).
2. `candidate`
   - Target: release-candidate and beta cohorts.
   - Version rule: prerelease versions containing `rc` or `beta` (for example `1.5.0-rc.1`).
3. `canary`
   - Target: short-lived pre-validation builds.
   - Version rule: prerelease versions containing `canary`, `alpha`, or `dev`.

## Upgrade Policy

1. `stable` upgrades:
   - allow patch/minor upgrades after checksum and signature verification.
   - major upgrades require explicit release-note acknowledgment.
2. `candidate` upgrades:
   - manual opt-in only.
   - validate against release evidence and unresolved blocker list before rollout.
3. `canary` upgrades:
   - manual opt-in only.
   - no automatic promotion to stable; must pass candidate/stable gates first.

## Rollback Policy

1. Roll back by redeploying the previous signed artifact from the same channel.
2. Do not bypass signature validation during rollback.
3. Record rollback trigger, owner, and timestamp in release evidence.

## Deprecation Policy

1. Keep `candidate` and `canary` artifacts for one minor cycle after stable promotion.
2. Deprecate older artifacts by marking them superseded in release notes and evidence.
3. Do not remove evidence/signatures for deprecated artifacts.

## Signature and Integrity Requirements

1. Every distributed artifact must have:
   - SHA256 manifest (`release.sha256`)
   - detached signature (`release.sha256.asc`)
2. Workflow must fail on missing signing key material.
3. Workflow must verify `gpg --verify artifacts/release.sha256.asc artifacts/release.sha256` before publish.

## Workflow Enforcement

1. `forge-release.yml` validates release evidence and sign-offs.
2. `forge-release.yml` validates channel/version compatibility and emits `artifacts/distribution-manifest.json`.
3. Release is blocked if policy validation or signature verification fails.

## Operator Checklist

1. Choose channel (`stable`, `candidate`, `canary`).
2. Confirm version matches channel policy.
3. Run required technical gates from `docs/release-checklist.md`.
4. Verify release evidence references blockers, owners, and dates.
5. Verify checksums/signatures before distribution.
